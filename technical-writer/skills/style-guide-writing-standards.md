# Style Guide & Writing Standards
Establish and enforce documentation style standards:

### Core Style Principles
1. **Be concise**: Use short sentences. Remove unnecessary words. "In order to" → "To".
2. **Use active voice**: "Click the button" not "The button should be clicked".
3. **Use present tense**: "The function returns" not "The function will return".
4. **Use second person**: "You can configure" not "The user can configure" or "One can configure".
5. **Be specific**: "Enter your API key" not "Enter the appropriate credentials".
6. **Be consistent**: If you call it a "dashboard" on one page, don't call it a "control panel" elsewhere.

### Formatting Standards
| Element | Convention |
|---------|------------|
| UI elements | **Bold** ("Click **Save**") |
| Code elements | `backticks` ("Set `timeout=30`") |
| File paths | `code format` ("`/etc/config.yaml`") |
| Variables | UPPER_SNAKE_CASE for env vars, code format for others |
| Key terms (first use) | *Italics* with a definition |
| Procedures | Numbered steps |
| Options/lists | Bullet points |

### Inclusive Language
- Avoid: master/slave, whitelist/blacklist, sanity check, guys.
- Use: primary/replica, allowlist/blocklist, check/verify, everyone/team.
