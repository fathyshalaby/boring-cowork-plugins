# Documentation Architecture
Design effective documentation systems and information architecture:

### Documentation Types
| Type | Audience | Purpose | Examples |
|------|----------|---------|----------|
| Tutorials | Beginners | Learning by doing | Getting started guide, "Hello World" |
| How-to Guides | Intermediate | Solving specific problems | "How to deploy to production" |
| Reference | All levels | Precise information lookup | API docs, CLI reference, config options |
| Explanation | Intermediate | Understanding concepts | Architecture overviews, design decisions |

This model (Diátaxis framework) helps structure documentation around user needs rather than product features.

### Information Architecture
- **Navigation**: Users should find what they need in ≤3 clicks.
- **Progressive disclosure**: Start with the simplest use case, then reveal complexity.
- **Consistent structure**: Every page of the same type follows the same template.
- **Search-first design**: Many users arrive via search, not navigation. Ensure every page stands alone with sufficient context.

### Content Strategy
- Inventory existing docs: What exists? What's outdated? What's missing?
- Prioritize by user impact: What questions does support answer most?
- Establish ownership: Every doc has an owner responsible for keeping it current.
- Review cycle: Set a review schedule (quarterly for critical docs, annually for stable features).
